## Session 2 — 13:00–13:45Z — Fixed two independent analyze failures (504 masking + prose-wrapped JSON)

Ross reported meal entry erroring again on 2026-07-09. Not a retired model this time — two separate bugs, both reproduced live and both now fixed and verified on production.

### What we did
- Reproduced the failure against the live endpoint: web-search-heavy meals exceed Netlify's ~30s function ceiling, Netlify returns an **HTML** error page, and `res.json()` (called before `res.ok`) threw a JSON syntax error that masked the real 504. Measured: 1 of 6 realistic meals timed out at 30.3s.
- Found a **second, pre-existing bug** while verifying the first: the model routinely prefaces its JSON with prose ("Based on known nutritional data…"), so `JSON.parse` of the last text block threw. Confirmed it fails 2/3 runs on the *pre-change* settings, so it was independent of the timeout fix and not introduced by it.
- Fixed both in `index.html` (`2a82508`): added `readAnalyzeResponse()` (text-first parsing, maps a non-JSON body to the HTTP status) and `extractJson()` (scans text blocks newest-first, slices first `{` to last `}`). Cut `max_uses` 3→1, raised `max_tokens` 1000→1500. Deployed and verified 8/8 live, including the two previously-failing meals.

### Decisions made
- Treat `stop_reason: max_tokens` as a *fallback explanation* after extraction fails, not a pre-emptive veto — checking it up front would falsely reject complete responses that happen to end at the cap.
- Reduced `max_uses` to 1 rather than raising the Netlify function timeout: one search is what the successful cases actually used, and it needs no dashboard change. If timeouts return, raising the function timeout is the next lever.
- Appended a dated correction to `2026-07-08.md` rather than rewriting it — historical notes are a record of what was believed, not a wiki.
- Did **not** add request logging to `analyze.mjs`. Offered it to Ross (would make the next failure diagnosable from real input) but it puts meal descriptions in Netlify logs. Awaiting his call.

### Gotchas surfaced
- **A serverless error response is not JSON.** Netlify serves `text/html` on a 504. `await res.json()` before checking `res.ok` destroys the real status — the same masking pattern as the 2026-07-08 model retirement, where the true error existed but the client ate it.
- **An LLM text block is not pure JSON**, no matter what the system prompt says. Prose prefixes and multi-block replies are routine once web search is on.
- **Latency is the hidden variable.** The model *chooses* whether to search, so visually similar meals differ 10x in duration (Chipotle bowl 2.2s / no search; Panera salad 17–24s / searches). That is why the errors felt random.
- **A memory note was wrong and nearly stopped the investigation.** `2026-07-08.md` claimed there's no local Netlify auth. The CLI is in fact authenticated (team 3SM). Corrected in `99354f5`.
