# 2026-07-08 — Fixed retired-model outage in meal analyze + added self-healing model fallback

## What we did
- Diagnosed why "Analyze Macros" was failing: the app pinned `claude-sonnet-4-20250514`, which Anthropic retired 2026-06-15 — retired IDs 404 with no `content`, so the frontend threw on `data.content.filter(...)` and showed a generic "try again."
- Swapped both analyze flows to `claude-sonnet-5` (thinking disabled to protect the small `max_tokens` budget; web-search bumped to `web_search_20260209`), and made the error toast surface the real API message instead of crashing silently.
- Made it durable: moved model choice out of `index.html` into `analyze.mjs` as a fallback chain `[claude-sonnet-5, claude-opus-4-8]` that falls through on a retirement 404; verified the fallback fires live by deploying with a retired model first (served-by came back `claude-sonnet-5`), then landed the clean list.

## Decisions made
- Chose the Anthropic two-model fallback over an OpenRouter migration: OpenRouter buys cross-provider redundancy but is a rewrite (OpenAI-shaped responses + different web-search mechanism) and needs a new Netlify secret; the fallback solves the stated retirement problem with far less risk. Ross explicitly picked "easy / your recommendation."
- Rejected runtime model auto-discovery (Models API) — a future model could change the request surface and break in a new way; a short verified list is safer.
- Model list is overridable without a code deploy via the `ANALYZE_MODELS` Netlify env var (comma-separated, highest priority first).

## Open threads / next steps
- The fallback list is not infinite — both models eventually retire (~yearly). Refresh `MODELS` in `analyze.mjs` (or set `ANALYZE_MODELS` in the Netlify dashboard) every couple years. A single retirement is now a non-event.
- Not browser-UI-verified: all checks were direct `curl` to `/api/analyze` and the served HTML. The API layer that was broken is confirmed working; the button-tap → macros-render → Pabbly/sheet-write path is inferred. Ross to spot-check with a real meal entry if he wants button-level certainty.

## Gotchas surfaced
- Anthropic retires model snapshots ~yearly; a hard-pinned model ID silently expires and 404s. The frontend swallowed the error, making it look like an app bug rather than a dead model — the error-surfacing fix makes the next one diagnosable at a glance.
- No local Anthropic/OpenRouter usable path for this app: the deployed function's key lives only in Netlify env (macro-tracker is a documented Netlify-secrets exception), and there's no `ant` CLI or Netlify auth token locally — so live verification must go through the deployed endpoint, not a local call.
